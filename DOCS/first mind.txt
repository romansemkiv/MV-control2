ПЗ з серверним розгортанням в докері. 
Система дозволяє користувачам гнучко і швидко налаштовувати візульний вигляд мультивювера і роутити на нього певні джерела.

Клієнти працюють через WebGui. 
Існує система ролей - admin/user. Мінімальна база данних для збереження користувацьких налаштувань, проста система автентифікації. 

Користувач бачить в своєму акаунті виключно ті джерела (Sources) та мультивювери, які дозволив адміністратор.
Користувач може:
Бачити актуальний роутинг (Sources)
Бачити актуальний стан мультивюверів
Змінювати роутинг для своїх MV
Змінювати Layout в MV
Редагувати UMD
Міняти відображення PCM Audio bars
Використовувати функцонал пресетів. 

Система збереження пресетів.
Може
Зберігати, викликати, скачувати, завантажувати пресети.
Система має виглядати як окреме вікно розділене на дві частини, в верхній вибір мултивювера/мультивюверів для збереження
в нижній частині вибір параметрів які будуть збережені, система чекбоксів. 
Такіж опції мають бути доступні як при збережені так і при виклику параметрів. 
Пресети зберігаються в системі, доступні тільки користувачеві, з можливістю скачувати та завантажувати в систему. 

Адміністратор має:
Управляти користувачами - створення, видалення, назначення прав (admin/user), скидання пароля.
Управління параметрам для протоколами Quartz та NEXX-api.
Налаштування параметрів для підключень.
Назначати доступні Sources та мультвювери для кожного користувача.
функціонал користувача. 

Система використовує два протоколи від Evertz: Quartz для отримання актуального стану роутингу та відпраки команд переключення - працює з Magnum сервером, NEXX-api для зяитування та управління параметрами мультивювера - працює з роутером NEXX. 


NEXX-API
Отримує актуальний стан, відправляє команди для зміни параметрів. 
1 пристрій може містити певну кількість мультивюверів. Кількість береться в запиті, параметр Licensed Multiviewers
1 мультивювер може містити до 16 вікон. Це стандарт, але фактична кількість залежить від конкретного Layout.
Кожен мультивювер має набір параметрів які застовуються до всіх данних в ньому:
	- Text font
	- Outer Border Pixels
	- Inner Border Pixels
	- Layout
В системі (NEXX) є зашито 43 Layout. Кожен Layout містить різне положення і різну кількість вікон.

Кожне вікно має свій номер. 
Кожне вікно може містити 3 UMD текста різного типу та 1 PCM Audio Waform.

Кожен PCM Audio Waform може відображати різну кількість PCM Audio bars - 0, 2, 4, 6, 8, 12, 16.

Кожен з 3х UMD має свої параметри
- тип (Off, static, Dynamic line 1, NTP time, NTP time w offset)
- UMD text (тільки для static)
- UMD Box color
- UMD BOX Alpha
- UMD BOX position (X/Y)
- UMD Text Colour
- UMD Text Alpha
- UMD Text Size
- UMD Padding

Данні в протоколі передаюються через Enum списки. Потрібне створення карти. 

Quartz
Роутинг відбувається шляхом назначення доступного для користувача Source в конкретне window. 
Протокол Quartz має певні обмеження.
К-сть Sources задається глобально і lable відповідає номеру input.
з Destination все складніше так як потрібно замапити Destination з протоколу Quartz з номерами window в мультивювер в додатку
Буде наступна схема. Адмін задає кількість мультивюверів, кожному мультивюверу відповідає 16 каналів output. І мапиться mv1.1 - out 1, mv2.1 - out 17 і тд... 
 
Серверний бекенд розвертається через portainer прямо з репощтроію GitHub, збірка відбувається прямо в doker, архітектура x86. 
автоматичне оновлення деплою через webhook. 

Backend
Після зчитування параметри зберігаються в БД. Оновлення відбувається 1 раз на добу, або за вимогою користувача - кнопка update.
Зміни застосовуються автоматично, але з певним захистом від масового оновлення. Протоколи досить важкі і повільні. 
Перший користувач в системі автоматично стає адміном. Вимога встановити пароль при першому логіні. Системи відновлення паролів немає. Тільки скидання адміном.
